\# Project Understanding - 王振寰



\## 1. 我对项目目标的理解



我理解这个 Multi-Agent 项目管理系统的目标，不是单纯做一个聊天机器人，而是围绕项目管理流程构建一个能够协助团队完成任务发布、任务提交、内容汇总、Agent 评估和教师审核的系统。



从学习小组文档来看，这个系统的核心场景是数据结构课程中的作业发布、提交和辅助批改流程。系统需要模拟教师和学生两个角色：教师可以通过 Mattermost 或前端页面发布作业、查看学生提交内容、获取 Agent 生成的初步评估报告，并对评估结果进行确认或修改；学生可以提交作业答案，系统负责收集、整理和展示这些提交内容。



因此，这个项目的重点不是单点功能，而是把协作平台入口、后端 API、数据库结构、前端 Dashboard 和 AI Agent 能力串成一个完整流程。它更接近一个“AI 辅助的项目 / 作业管理系统”，而不是普通的任务列表工具。



\## 2. 我对整体链路的理解



我理解的系统整体链路大致如下：



```text

Mattermost / 前端页面

↓

后端 API

↓

数据库

↓

Agent 服务

↓

评估报告 / 汇总结果

↓

教师审核与反馈

```



其中，Mattermost 主要作为协作入口，可以通过 Slash Command、Webhook 或 Bot 的形式接收用户操作。例如教师可以在 Mattermost 中发布作业，学生可以提交内容，系统可以把处理结果再推送回 Mattermost。



后端 API 负责接收请求、校验参数、处理业务逻辑，并与数据库和 Agent 服务交互。数据库需要保存用户、作业、提交内容、评估结果、评论记录等结构化数据。Agent 服务负责对学生提交内容进行初步分析，例如检查答案完整性、判断主要问题、给出修改建议或基础评分。



前端 Dashboard 则用于展示系统状态，例如作业列表、提交记录、评估结果、审核状态等，让教师或管理员可以更直观地查看和操作。



\## 3. 我对 MVP 范围的理解



我认为第一阶段 MVP 不需要一次性实现完整系统，而应该优先跑通一个最小闭环。



我理解的 MVP 可以包括以下功能：



1\. 教师发布作业

&#x20;  教师可以通过 Mattermost Slash Command 或前端页面发布一份作业，作业信息至少包括标题、内容描述和截止时间。



2\. 学生提交作业

&#x20;  学生可以通过系统提交对应作业的答案。提交内容可以先用文本形式模拟，不一定一开始就支持复杂文件上传。



3\. 系统汇总提交内容

&#x20;  后端能够从数据库中查询某个作业下所有学生的提交内容，并形成基础汇总。



4\. Agent 生成初步评估报告

&#x20;  系统可以调用 AI Agent，对学生提交内容生成简单评估结果，例如答案完整性、主要问题、修改建议和基础评分。



5\. 教师查看和审核结果

&#x20;  教师可以查看 Agent 的评估结果，并进行确认、修改或要求重新评估。



6\. 基础前端展示

&#x20;  前端页面能够展示作业、提交内容、Agent 评估结果和审核状态。



我认为 MVP 的核心不是功能数量多，而是要证明这条链路能跑通：



```text

作业发布 → 学生提交 → 系统汇总 → Agent 评估 → 教师审核

```



\## 4. 我认为需要重点学习的技术模块



结合项目要求和我目前的能力，我认为接下来需要重点学习以下内容。



\### 4.1 Git 协作



需要熟悉 clone、branch、commit、push、pull request、code review 等基本流程。这个作业本身就是一次 Git 协作训练，目的是让成员适应正式项目开发中的协作方式。



\### 4.2 Mattermost 集成



需要理解 Mattermost 在系统中的入口作用，包括 Slash Command、Webhook、Bot Account 和消息推送。现阶段我需要先理解 Slash Command 如何把用户输入发送到后端，以及后端如何返回响应。



\### 4.3 后端 API 与数据库建模



需要理解 REST API 的基本设计方式，以及如何围绕 Project、User、Assignment、Submission、Comment、Evaluation 等对象建模。后端不仅要能接收请求，还要能把协作平台中的非结构化操作转化为数据库中的结构化记录。



\### 4.4 前端 Dashboard



需要掌握 React / TypeScript 的基础用法，能够实现作业列表、提交列表、状态展示、按钮操作和 API 调用。前端不一定要一开始做得很复杂，但需要能够清晰展示系统流程。



\### 4.5 AI Agent 接入



需要理解 LLM API、Prompt 设计、结构化输出、JSON 校验和 Function Calling / Tool Calling 的基本概念。Agent 不应该只是随便生成一段文字，而应该输出可被系统使用的结构化评估结果。



\### 4.6 本地部署与环境管理



需要熟悉 Docker、Docker Compose、环境变量和数据库连接配置。正式项目中，成员需要能够在本地把最小系统跑起来，而不是只看代码。



\## 5. 我已经做过的初步实践



在正式完成作业前，我已经尝试做了一个本地 mini practice 项目，用来理解项目可能涉及的技术链路。这个练习项目包括：



\* 使用 Next.js / React / TypeScript 实现简单任务管理 Dashboard；

\* 使用 NestJS 实现后端 REST API；

\* 使用 PostgreSQL 和 Prisma 实现任务数据持久化；

\* 使用 Docker Compose 启动本地 PostgreSQL 数据库；

\* 模拟 Mattermost Slash Command 接口，通过命令创建任务和查看任务统计；

\* 为 Mattermost 模拟接口加入 token 校验；

\* 使用 Git 和 GitHub 完成 commit、push 和 README 文档整理。



通过这个练习，我对“前端页面 → 后端 API → 数据库 → 协作平台命令入口”的基本链路有了更具体的理解。不过这个练习只是一个简化版本，距离正式项目还需要继续补充用户角色、作业模型、提交记录、Agent 评估和教师审核等功能。



\## 6. 我目前的疑问



目前我还有以下几个疑问，希望后续在学习小组或正式开发中逐步明确：



1\. 第一阶段 MVP 中，Mattermost 是必须作为主要入口，还是可以先用前端页面模拟入口？

2\. 教师和学生的身份是否需要在第一阶段就实现登录和权限管理，还是可以先用简单字段模拟？

3\. 学生提交内容的形式是纯文本、代码片段，还是需要支持文件上传？

4\. Agent 评估报告是否需要固定 JSON 结构？如果需要，具体字段和评分标准应该如何定义？

5\. 教师审核 Agent 结果时，需要保留修改前后的历史记录吗？

6\. 数据库模型是否已经有推荐设计，还是需要学习小组先自行提出初版？

7\. Mattermost 的真实接入是否需要本地部署 Mattermost，还是可以先通过模拟 Slash Command 请求完成？

8\. 最终验收更看重功能完整性、代码质量，还是对项目链路的理解和可展示 Demo？
