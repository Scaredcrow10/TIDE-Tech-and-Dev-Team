## 1. 我理解的项目目标

> 我理解这个项目当前的目标是构建一个 **Multi-Agent 项目管理系统**，第一阶段聚焦于面向数据结构课程的作业批改场景。系统通过 Mattermost 作为协作入口，后端对接 AI Agent，实现从作业发布、学生提交、汇总到 Agent 自动评估、教师审核的完整流程。项目最终希望降低教师重复性批改工作负担，同时为数据结构课程提供一套可运行、可说明的智能化教学辅助工具。

项目面向的主要用户是：

- 数据结构课程的**教师**（发布作业、审核评估结果）
- 数据结构课程的**学生**（提交作业、接收反馈）

最终价值：

- 将自然语言交互（Mattermost 聊天）与结构化任务管理打通
- 利用 AI Agent 提供基础评估能力，辅助教学决策
- 形成一套可在本地完整运行的 MVP 系统

## 2. 我理解的整体链路

### Mattermost 本身是什么

通过阅读 Mattermost 源码仓库（`mattermost/mattermost`），我了解到 Mattermost 是一个 **Go + React 单体仓库**，核心结构如下：

```
mattermost/
├── server/                    # Go 后端
│   └── channels/
│       ├── api4/              # REST API v4（约80+个 handler 文件，扁平结构）
│       │   ├── bot.go         # Bot Account CRUD
│       │   ├── agents.go      # AI Agent 相关端点
│       │   ├── command.go     # Slash Command 注册与管理
│       │   ├── channel.go     # 频道管理
│       │   ├── webhook.go     # 入站/出站 Webhook
│       │   ├── integration_action.go  # 交互式按钮/菜单
│       │   └── ...
│       ├── app/               # 业务逻辑层（模块化目录）
│       │   ├── slashcommands/ # 33+ 内置命令（/invite, /away 等）
│       │   ├── teams/         # 团队管理
│       │   ├── users/         # 用户管理
│       │   └── ...
│       ├── store/             # 数据访问层（对接 PostgreSQL）
│       └── model/             # 数据模型定义
├── webapp/                    # React/TypeScript 前端
│   ├── channels/              # 主聊天界面
│   └── platform/              # 共享组件库
└── enterprise/                # 企业版扩展（独立于开源核心）
```

关键发现：**Mattermost 已经有 AI Agent 基础设施**，三个端点直接暴露：

- `GET /api/v4/agents` — 列出可用 Agent
- `GET /api/v4/agents/status` — AI Plugin Bridge 状态
- `GET /api/v4/llmservices` — 列出 LLM 服务

这意味着我们的系统**不是从零造轮子**，而是站在 Mattermost 已有能力之上。

### 关键交互流程

1. **教师发布作业** → Mattermost 中发送 Slash Command 或通过 Bot 交互 → 通过 Webhook 转发到我们的后端 → 存入 PostgreSQL
2. **学生提交作业** → 在 Mattermost 频道或前端页面提交 → 后端记录提交内容
3. **系统汇总** → 后端按作业维度聚合所有学生提交 → 通过 Mattermost Bot 消息推送给教师
4. **Agent 评估** → 后端调用 AI Agent 服务 → Agent 调用 LLM API → 返回结构化评估报告 → 存入数据库
5. **教师审核** → 在前端 Dashboard 或 Mattermost 中查看评估结果 → 确认/修改/重新评估 → 反馈通过 Bot 发回

## 3. 我理解的 MVP 范围

- MVP 应该包含：
  - 教师通过 Mattermost 发布数据结构作业
  - 学生提交作业答案（文本/代码）
  - 系统按作业维度汇总所有提交
  - AI Agent 对每份作业生成基础评估报告（完整性、正确性、问题说明、建议、评分）
  - 教师对评估结果进行确认、修改或要求重新评估
  - Docker Compose 一键启动所有服务

- MVP 暂时不需要包含：
  - 完整的用户权限系统（角色模拟即可）
  - 多课程/多学期管理
  - 实时消息推送通知
  - 复杂的评分统计和数据分析
  - 云部署（本地运行即可）

- 我认为优先级最高的是：
  1. 完整链路跑通（Mattermost → 后端 → DB → Agent → 前端）
  2. Agent 评估质量（Prompt 设计和结构化输出校验）
  3. 教师审核流程（人机协同的最后一环）

## 4. 我存在的疑问

- 大作业中，我们应该走 Mattermost **Plugin** 路线还是**外部服务 + Webhook** 路线？Plugin 的好处是能直接调用 Mattermost 内部 API（无需 HTTP），但需要 Go 编写；外部服务可以用 TypeScript 统一技术栈。执行方案没明确指定，管理者有偏好吗？

- Mattermost 已有的 `agents.go` 和 `/api/v4/llmservices` 看起来是面向企业版的 AI 功能（Copilot 之类），我们的 Agent 是复用这套基础设施，还是独立实现一个 Agent 微服务？如果是后者，两者如何区分？

- 作业 5 的 Agent 任务提取和最终大作业的 Agent 评估批改，本质上是两个不同的 Prompt 场景——一个是从聊天记录提取任务，一个是评估代码正确性。这两个 Agent 在最终系统里是**同一个服务内的两个 Prompt 模板**，还是**两个独立的 Agent 微服务**？

- 大作业的"数据结构题目数据集"由管理者提供（7.20 前发布），但 Agent 的评估 Prompt 需要我们自己设计。对于数据结构这种有确定性答案（如算法复杂度）的题目，如何设计 Prompt 让 LLM 既能识别完全正确的答案，又能对部分正确给出合理评分，而不只是"看起来对就给过"？是否需要参考数据（标准答案/评分标准）作为 Few-shot 输入？
